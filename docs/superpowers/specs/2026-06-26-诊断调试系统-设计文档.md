# 诊断调试系统 — 设计文档

> 版本：v1.0  
> 日期：2026-06-26  
> 状态：设计已确认，待实施

---

## 一、背景与目标

### 1.1 问题

学生会线上交流平台包含 8 个业务模块，随着代码量增长，后期出现 bug 时面临以下问题：

- 错误堆栈信息有限，难以快速判断出问题的是哪个模块/文件
- API 调用失败时缺乏上下文，不知道是哪个模块的哪个请求
- 每次排查需要遍历整个代码库，效率低

### 1.2 目标

建立一套轻量的结构化诊断标签系统，实现：

1. **错误精准定位**：每个错误自动带上 `[模块/组件]` 标签，搜标签即定位到文件
2. **API 调用追踪**：每个 Supabase 请求自动记录耗时和结果，快速定位后端相关问题
3. **模块化、低侵入**：不破坏现有模块结构，不改 UI，对业务代码的影响最小化
4. **预留扩展空间**：接口化设计，后期可接入外部监控服务（Sentry 等）而无须改动已有代码

---

## 二、方案概述

采用「轻量 Logger + 结构化标签」方案：

- 新建独立诊断基础设施层 `src/diagnostics/`
- 提供带模块标签的结构化日志函数
- 增强现有 ErrorBoundary 集成日志输出
- 提供 Service Wrapper 自动追踪 API 调用
- 生产环境自动降级日志级别

### 核心决策

| 维度 | 决策 |
|------|------|
| 方案类型 | 轻量 Logger（方案1，手动标注） |
| 标签粒度 | 模块 + 组件级（定位到文件，如 `tasks/TaskForm`） |
| 覆盖范围 | 错误捕获 + API 调用追踪 |
| 扩展策略 | 接口抽象 + 配置开关 + 预留 errorReporter |

```
│   ├── index.ts                  ← 对外入口，按需导出
│   ├── types.ts                  ← 类型定义（ModuleTag、LogLevel、LogEntry、IErrorReporter）
│   ├── config.ts                 ← 全局配置（日志级别、输出目标、环境切换）
│   ├── logger.ts                 ← 核心：结构化日志引擎
│   ├── serviceWrapper.ts         ← Service 调用追踪包装器
│   └── errorReporter.ts          ← 错误收集与聚合接口（本次实现基础版，预留远程上报）
│
├── components/
│   ├── ErrorBoundary.tsx          ← 增强：componentDidCatch 集成 logger
│   └── ModuleErrorBoundary.tsx    ← 增强：componentDidCatch 集成 logger + 模块名
│
├── modules/                       ← 不改目录结构
│   └── <各模块>/
│       ├── index.tsx              ← 不变
│       ├── *Service.ts            ← 使用 trace() 包装 API 函数
│       └── *.tsx                  ← 组件内按需使用 logger
│
└── utils/
    └── constants.ts               ← 新增：MODULE_TAGS 标签常量定义
```

### 分层原则

1. **`diagnostics/` 是纯基础设施层**，不 import 任何业务模块
2. **模块单向依赖 diagnostics**：`modules/*` → `diagnostics/`，绝不反向
3. **每个文件职责单一**：logger 只管输出，wrapper 只管包装，config 只管配置
4. **入口收敛**：模块通过相对路径 `import { logger } from '../../diagnostics'` 使用，内部实现可替换

## 四、核心模块设计

### 4.1 types.ts — 类型定义

```ts
// 日志级别
export type LogLevel = 'debug' | 'info' | 'warn' | 'error';

// 模块标签格式：模块名/文件名
// 示例：'tasks/TaskForm' | 'tasks/taskService' | 'auth/LoginPage'
export type ModuleTag = string;

// 单条日志结构
export interface LogEntry {
  timestamp: number;        // Date.now()
  level: LogLevel;
  tag: ModuleTag;
  message: string;
  data?: Record<string, unknown>;
}

// 错误报告接口（预留外部监控接入）
export interface IErrorReporter {
  report(error: Error, tag: ModuleTag, context?: Record<string, unknown>): void;
}

// Logger 实例接口（预留替换实现）
export interface ILogger {
  debug(message: string, data?: Record<string, unknown>): void;
  info(message: string, data?: Record<string, unknown>): void;
  warn(message: string, data?: Record<string, unknown>): void;
  error(message: string, error?: Error | unknown, data?: Record<string, unknown>): void;
  api(method: string, success: boolean, durationMs: number, data?: Record<string, unknown>): void;
}

// 诊断系统配置
export interface DiagConfig {
  level: LogLevel;
  target: 'console' | 'remote';
  errorReporter?: IErrorReporter;
}
```

### 4.3 logger.ts — 结构化日志引擎

核心设计：

```ts
import { diagConfig } from './config';
import type { ILogger, LogLevel, ModuleTag } from './types';

const LEVEL_ORDER: Record<LogLevel, number> = {
  debug: 0, info: 1, warn: 2, error: 3,
};

// 判断当前级别是否应该输出
function shouldLog(level: LogLevel): boolean {
  return LEVEL_ORDER[level] >= LEVEL_ORDER[diagConfig.level];
}

// 格式化输出到 console
function output(level: LogLevel, tag: ModuleTag, message: string, extra?: string) {
  const prefix = `[${level.toUpperCase()}] [${tag}]`;
  const text = extra ? `${prefix} ${message} ${extra}` : `${prefix} ${message}`;

  switch (level) {
    case 'error': console.error(text); break;
    case 'warn':  console.warn(text);  break;
    case 'info':  console.info(text);  break;
    default:      console.log(text);   break;
  }
}

// 创建绑定标签的 logger 实例
function createLogger(tag: ModuleTag): ILogger {
  return {
    debug(msg, data)      { if (shouldLog('debug')) output('debug', tag, msg, data && JSON.stringify(data)); },
    info(msg, data)       { if (shouldLog('info'))  output('info',  tag, msg, data && JSON.stringify(data)); },
    warn(msg, data)       { if (shouldLog('warn'))  output('warn',  tag, msg, data && JSON.stringify(data)); },
    error(msg, err, data) {
      if (shouldLog('error')) {
        const errMsg = err instanceof Error ? err.message : String(err ?? '');
        output('error', tag, msg, data
          ? `{ error: "${errMsg}", ...${JSON.stringify(data)} }`
          : `{ error: "${errMsg}" }`);
        // 同时调用 errorReporter（如果已配置）
        if (err instanceof Error) diagConfig.errorReporter?.report(err, tag, data);
      }
    },
    api(method, success, durationMs, data) {
      if (shouldLog('info')) {
        const status = success ? '完成' : '失败';
        output('info', tag, `${method} ${status} { duration: ${durationMs}ms }`,
          data ? ` ${JSON.stringify(data)}` : '');
      }
    },
  };
}

// 对外：通过标签获取 logger
export const logger = {
  for: createLogger,
};
```

```
[INFO]  [tasks/taskService] createTask 完成 { duration: 234ms }
[WARN]  [notices/NoticeForm] 响应超时，正在重试

```

### 4.4 serviceWrapper.ts — API 调用追踪

```ts
import { logger } from './logger';
import type { ModuleTag } from './types';

// 包装单个 async 函数，自动追踪耗时和结果
export function trace<T extends (...args: any[]) => Promise<any>>(
  tag: ModuleTag,
  methodName: string,
  fn: T,
): T {
  const log = logger.for(tag);
  return (async (...args: any[]) => {
    const start = performance.now();
    try {
      const result = await fn(...args);
      log.api(methodName, true, Math.round(performance.now() - start));
      return result;
    } catch (err) {
      log.api(methodName, false, Math.round(performance.now() - start));
      log.error(`${methodName} 执行失败`, err);
      throw err;  // 原样抛出，不吞错误
    }
  }) as T;
}
```

**设计要点：**
- 不改原函数签名，返回值和异常照常传递
- 使用 `performance.now()` 精确计时
- 错误照样抛出，调用方的 try-catch 不受影响
- 不做重试（重试属于 service 自身职责）

### 4.5 errorReporter.ts — 错误收集（预留扩展）

```ts
import type { IErrorReporter, ModuleTag } from './types';
import { diagConfig } from './config';

// 基础实现：收集到内存数组（最多保留最近 50 条，避免内存泄漏）
const MAX_ERRORS = 50;
const errorLog: Array<{ time: number; tag: ModuleTag; message: string }> = [];

export const consoleErrorReporter: IErrorReporter = {
  report(error: Error, tag: ModuleTag, context?: Record<string, unknown>) {
    errorLog.push({ time: Date.now(), tag, message: error.message });
    if (errorLog.length > MAX_ERRORS) errorLog.shift();
    // 预留：后期替换为 Sentry.captureException(error, { tags: { module: tag }, extra: context })
  },
};

// 获取最近错误列表（供调试面板使用，预留）
export function getRecentErrors() {
  return [...errorLog];
}

// 初始化：注入 errorReporter
export function initErrorReporter(reporter?: IErrorReporter) {
  diagConfig.errorReporter = reporter ?? consoleErrorReporter;
}
```

## 七、使用示例

### 场景一：渲染报错

控制台输出：
```
[ERROR] [tasks/TaskListPage] 模块渲染错误 { error: "Cannot read properties of null", module: "任务管理" }
```
开发者行为：搜索 `tasks/TaskListPage` → 直接打开 `src/modules/tasks/TaskListPage.tsx` → 检查渲染逻辑。

### 场景二：API 调用失败

控制台输出：
```
[INFO]  [tasks/taskService] createTask 失败 { duration: 1200ms }
[ERROR] [tasks/taskService] createTask 执行失败 { error: "Request timeout" }
```
开发者行为：知道是 `taskService` 的 `createTask` 超时 → 检查 Supabase 连接或网络。

### 场景三：全局崩溃

控制台输出：
```
[ERROR] [app/ErrorBoundary] 全局渲染错误 { error: "ChunkLoadError", stack: "..." }
```
开发者行为：知道是代码分块加载失败 → 检查 Vite 构建配置或部署。
