## 一、背景与目标

### 1.1 一期回顾

一期（2026-06-29）已完成 5 项增强：

| 增强 | 状态 |
|------|------|
| 任务模板系统 + 交接备注 + 跨部门协作 | ✅ 已上线 |
| 论坛知识库结构化（会议纪要 / 活动复盘 / 外联通讯录） | ✅ 已上线 |
| 个人中心工作量热力图 + 排行榜 | ✅ 已上线 |
| 成员工作看板（主席视角） | ✅ 已上线 |
| 公告关联任务 | ✅ 已上线 |

### 1.2 二期痛点

一期解决了"知识沉淀"和"全局可见"的问题，但**新成员上手**和**部门负责人日常管理**仍存在盲区：

| 痛点 | 表现 | 根因 |
|------|------|------|
| 新人找不到人 | 不知道某件事该问谁、不知道对应部门的成员有哪些 | 没有集中的人员目录 |
| 新人不知道做到哪算完 | 有了模板步骤，但没有截止时间和检查点，执行靠自觉 | 步骤是静态文字，不是活跃的检查点 |
| 新人找不到组织规范 | 不知道找谁报到、常用哪些模板、常见问题怎么处理 | 知识散落在帖子中，没有集中的"部门指南" |
| 公告里的任务要手动抄 | 公告说了要做的事，需要手动去任务模块重新创建 | 公告和任务之间有间隔，多一步就多一个遗忘概率 |

### 1.3 目标角色

| 角色 | 核心诉求 |
|------|---------|
| **部门负责人** | 新人上手快、减少重复解释、日常管理省事 |
| **校方领导（团委老师/学工部）** | 合规留痕、责任可追溯、数据可考核 |
| **新入会志愿者** | 知道找谁、知道做什么、知道做到什么程度算完成 |

### 1.4 设计原则

延续一期：

| 原则 | 说明 |
|------|------|
| 模块不改结构 | 所有增强在已有文件内新增组件/函数，不动目录布局 |
| 不改路由 | 功能增强在已有页面内通过 Tab/Card/Modal 展开 |
| 兼容已有数据 | 新增列为可选字段，有默认值，不影响已有数据查询 |
| 沿用诊断体系 | 所有新增代码使用 `logger.for(标签)`，搜标签定位文件 |
| 不改 UI 主题 | 延续 Ant Design 深蓝办公风 |
| xhigh 推理强度 | 所有子代理使用 xhigh 推理强度 |
| 子代理驱动开发 | 功能分解为独立任务，每个任务由独立子代理实现 |

## 二、功能一：智能通讯录

### 2.1 解决的问题

新人不知道找谁：不知道某件事问哪个部门、不知道那个部门的人叫什么、不知道怎么联系。

### 2.2 功能要点

**入口：** ProfilePage 内，统计卡片 + 热力图下方，新增「📇 通讯录」Card。

**搜索与筛选：**
- 顶部搜索框，支持按姓名、部门名、角色名模糊搜索
- 左侧部门列表（12 个部门），点击筛选该部门所有成员
- 搜索结果实时过滤，不需要额外请求

**成员卡片：**
- 网格布局（3-4 列），每张卡片显示：头像、姓名、部门、角色 Tag
- 当前任务数摘要：🟢 进行中 N 个 / 🔴 逾期 N 个
- 点击卡片展开详情 Panel（或 Popover）：学号、联系方式（如有）、最近完成的 3 个任务

**权限设计：**
- 查看通讯录：全员可见（volunteer 也能看，方便跨部门协作）
- 联系方式字段：仅 dept_head+ 可见（保护隐私）
- 无编辑功能，数据来自 users 表和 tasks 表聚合

### 2.3 数据来源

- `users` 表全量查询（学生会规模 < 500 人，全量查询无性能问题）
- `tasks` 表按 `assigned_to` 聚合：`COUNT(*) WHERE status IN ('in_progress')` 和 `COUNT(*) WHERE status = 'overdue'`

### 2.4 数据库变更

无。复用已有表。

| 文件 | 变更类型 | 说明 |

| `src/modules/profile/MemberDirectory.tsx` | **新增** | 智能通讯录组件 |
| `src/modules/profile/ProfilePage.tsx` | 修改 | 引入 MemberDirectory |

## 三、功能二：任务里程碑 + 自动提醒

### 3.1 解决的问题

新人拿到任务不知道从哪步开始、不知道每步什么时候做完、没人盯就拖着。一期模板已有 `steps` JSONB，但这些步骤只是静态文字展示，没有截止时间和完成状态。

### 3.2 功能要点

**里程碑展示（TaskDetail 内）：**

在任务详情的步骤清单区域，升级为可交互的里程碑面板：

- 每个里程碑一行：☑ checkbox + 标题 + 截止时间 + 状态标签
- 面板顶部显示进度条（已完成数 / 总里程碑数）
- 任务执行者可以勾选完成里程碑
- 负责人/主席可以修改里程碑内容

**状态样式：**
- ✅ 已完成：绿色，显示完成时间和完成人
- ⚠️ 距截止 < 3 天：黄色高亮行
- 🔴 已逾期：红色高亮行 + 红色 Badge「逾期」
- ⚪ 待完成（距截止 > 3 天）：正常显示

**自动提醒机制（前端级，无后端定时任务）：**

| 触点 | 位置 | 提醒逻辑 |
|------|------|---------|
| 🔴 任务卡片红点 | TaskListPage | 有逾期里程碑 → 卡片标题旁显示红色 Badge「N 项逾期」 |
| ⚠️ 里程碑高亮 | TaskDetail | 距截止 < 3 天黄色，已逾期红色 |
| 📋 个人中心汇总 | ProfilePage | 统计卡片新增行：「⚠️ 里程碑逾期：N」「⏰ 近日截止：N」 |

**新建/编辑里程碑：**
- dept_head+ 发布任务时可勾选「启用里程碑模式」
- 若从模板创建且模板有 steps，自动转为里程碑（每步需手动填截止时间）
- 可在 TaskDetail 中随时添加/删除/编辑里程碑（dept_head+ 权限）
- 志愿者只能勾选完成，不能编辑里程碑

### 3.3 数据库变更

```sql
-- 新表：任务里程碑
CREATE TABLE IF NOT EXISTS task_milestones (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  task_id UUID REFERENCES tasks(id) ON DELETE CASCADE,
  title TEXT NOT NULL,
  description TEXT,
  deadline TIMESTAMPTZ,
  status TEXT NOT NULL DEFAULT 'pending',  -- pending | completed
  sort_order INTEGER DEFAULT 0,
  completed_at TIMESTAMPTZ,
  completed_by UUID REFERENCES users(id),
  created_at TIMESTAMPTZ DEFAULT now()
);

-- tasks 表加标记（区分是否启用里程碑模式）
ALTER TABLE tasks ADD COLUMN IF NOT EXISTS has_milestones BOOLEAN DEFAULT false;
```

## 四、功能三：部门新人指南

### 4.1 解决的问题

一期知识库模板提供了结构化的内容创建工具，但新人仍然不知道：
- 本部门的负责人和对接老师是谁
- 常用哪些模板
- 碰到常见问题怎么办（报销、请假、设备报修等）

信息散落在不同帖子里，没有集中的"部门入职页"。

### 4.2 功能要点

**入口：** ProfilePage 内通讯录卡片旁边，新增「🧭 部门新人指南」Card。

**内容结构（每部门一条记录）：**

```
┌─ 基本信息 ───────────┐
│ 部门负责人：余翰林      │
│ 对接老师：王老师        │
│ 办公地点：xxx          │
│ 部门群：xxxxxxxx       │
└──────────────────────┘
┌─ 常用模板 ───────────┐
│ 🔗 月度工作报告模板     │  → 链接到知识库帖子
│ 🔗 活动策划申请表       │
│ 🔗 会议室预约流程       │
└──────────────────────┘
┌─ 常见问题 FAQ ───────┐
│ ❓ 怎么报销活动经费？   │
│ → 找学办王老师，填报销单│
│ ❓ 打印机坏了找谁？     │
│ → 生活服务部，内线 3021 │
└──────────────────────┘
```

**编辑权限：**
- 查看：全员可见（volunteer 也能看自己部门的指南）
- 编辑：本部门 dept_head+ 可编辑自己部门的指南（Modal 表单）
- 老师/主席团可编辑所有部门指南
- 顶部下拉框可切换查看其他部门的指南

**初始填充：**
- 新表首次为空时，显示引导提示「本部门尚未编写新人指南，点击编辑开始创建」
- 不自动生成内容，由各部门负责人手动填写（保证信息准确）

### 4.3 数据库变更

```sql
-- 新表：部门指南（每部门一条记录，department 唯一）
CREATE TABLE IF NOT EXISTS department_guides (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  department TEXT UNIQUE NOT NULL,
  basic_info JSONB DEFAULT '{}',     -- { leader, teacher, office, group_chat }
  templates JSONB DEFAULT '[]',      -- [{ title, url }]
  faqs JSONB DEFAULT '[]',           -- [{ question, answer }]
  updated_by UUID REFERENCES users(id),
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

## 五、功能四：公告一键转任务

### 5.1 解决的问题

部门公告经常夹带"要做的事"（如"各部门请在周五前提交招新计划"），志愿者看到公告后需手动去任务模块创建任务。中间多一步，就多一个遗忘的概率。

### 5.2 功能要点

**触发按钮：**
- 公告卡片右下角新增「📋 转为任务」按钮
- 仅 dept_head+ 可见（公告面向全部门，但只有负责人能把公告内容转为正式任务并指派）

**转换 Modal：**

| 字段 | 默认值 | 说明 |
|------|--------|------|
| 任务标题 | 公告标题（可修改） | 自动填入，前缀加 `【公告任务】` 用于区分来源 |
| 截止时间 | 无默认值，必填 | 日期选择器 |
| 指派给 | 无默认值 | 可选本部门成员，留空 = 指派给全部门 |
| 优先级 | normal | 单选框：普通 / 重要 / 紧急 |
| 关联公告 | 当前公告 | 自动关联，不可修改 |

**创建后的关联效果：**
- 公告详情页「关联任务」区域（一期已实现）自动出现新创建的任务
- 任务详情页展示「📋 来源公告：xxx」链接，点击可跳转公告详情
- 任务卡片上不影响现有展示

### 5.3 数据库变更

```sql
-- tasks 表加公告来源字段
ALTER TABLE tasks ADD COLUMN IF NOT EXISTS linked_notice_id UUID REFERENCES notices(id);
```
