## 三、数据库设计

### 3.1 `usage_events` 表

```sql
CREATE TABLE usage_events (
  id         BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  event_type TEXT NOT NULL,
  user_id    UUID REFERENCES auth.users(id) ON DELETE SET NULL,
  module     TEXT,
  action     TEXT,
  metadata   JSONB DEFAULT '{}',
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_usage_type_time ON usage_events(event_type, created_at DESC);
CREATE INDEX idx_usage_user ON usage_events(user_id);
```

### 3.2 字段说明

| 字段 | 类型 | 说明 | 示例值 |
|------|------|------|------|
| `event_type` | TEXT | 事件大类 | `page_view`, `login`, `error`, `task_complete`, `notice_read`, `ticket_action` |
| `user_id` | UUID | 触发用户 | `ON DELETE SET NULL` — 用户删除后数据保留 |
| `module` | TEXT | 所属模块 | `tasks`, `notices`, `tickets`, `dashboard`, `forum` |
| `action` | TEXT | 具体动作 | `submitted`, `created`, `grabbed`, `returned` |
| `metadata` | JSONB | 附加信息 | `{"task_title":"迎新晚会","error_msg":"..."}` |

### 3.3 设计决策

- **JSONB 用于 metadata：** 不同事件类型存不同信息，灵活扩展
- **索引仅在 (event_type, created_at)：** 所有查询都是按类型 + 时间范围
- **不加 RLS：** 仅管理员通过 Service 层访问该表，不直接暴露给客户端

## 四、埋点工具函数

### 4.1 API

```typescript
// src/utils/analytics.ts

export type EventType = 'page_view' | 'task_complete' | 'notice_read'
  | 'ticket_action' | 'error' | 'login';

interface TrackPayload {
  event_type: EventType;
  userId: string;
  module?: string;
  action?: string;
  metadata?: Record<string, unknown>;
}

export function trackEvent(payload: TrackPayload): void;
```

### 4.2 设计要点

- **userId 由调用方传入：** 调用方已有 `user.id`（`useAuth()`），无需额外异步获取 session
- **fire-and-forget：** `.catch(() => {})` → Supabase 挂了也不影响用户操作
- **不引入诊断日志依赖：** 埋点本身不应触发诊断日志（避免循环依赖）
- **同步签名、异步执行：** 函数签名返回 `void`，内部 fire-and-forget

### 6.3 技术实现

- **统计卡片：** antd `Card` + `Statistic`，`Row` + `Col` 响应式（`xs={12} md={6}`）
- **排名列表：** antd `Table`（`showHeader={false}`, `pagination={false}`, `size="small"`）
- **错误日志：** antd `Table`
- **数据查询：** `adminService.ts` → `fetchAnalyticsSummary()`，内部 `Promise.all` 并行 4 个查询

### 6.4 Service 函数

```typescript
// adminService.ts 新增
export async function fetchAnalyticsSummary(): Promise<{
  totalEvents: number;
  activeUsers: number;
  recent7d: number;
  topModule: string;
  pageRanking: { module: string; count: number }[];
  eventStats: { event_type: string; count: number }[];
  recentErrors: { created_at: string; module: string; metadata: unknown }[];
}>
```

- 查询 `usage_events` 表
- 4 个并行查询（`Promise.all`）
- 默认 LIMIT：pageRanking 20 条，recentErrors 10 条
- error 为空时返回空数组（不抛异常）

## 九、性能影响

- **写入路径：** 每次埋点 1 次 Supabase INSERT（异步 fire-and-forget）→ 对用户体验影响为 0
- **查询路径：** 看板页面 4 次并行查询（`Promise.all`）→ 数据量小（<1M 行），<500ms
- **数据库增长：** 每次事件约 200 字节，预估每月 <1MB → 不需要数据清理策略（至少 3 年内不用管）
