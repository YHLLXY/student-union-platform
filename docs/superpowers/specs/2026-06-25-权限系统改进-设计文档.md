# 学生会交流平台 — 权限系统改进设计文档

> 状态：已确认。基于 2026-06-25 讨论。

## 一、背景

当前系统有 12 个部门、4 种角色层级，邀请码只关联部门不关联角色，所有新用户默认 `volunteer`。需要精细化权限控制，实现「邀请码决定一切」。

## 二、目标

1. 邀请码同时携带部门 + 角色信息，注册时自动分配
2. 教师使用独立登录入口
3. 主席/老师具备人员裁撤和部门调动权限
4. 权限层级清晰，模块边界明确

## 三、角色体系

| 级别 | 角色 | key | 所属部门 | 核心权限 |
|---|---|---|---|---|
| 0 | 常驻志愿者 | `volunteer` | 11 个普通部门 | 查看、提交、发帖、抢票 |
| 1 | 部门负责人 | `dept_head` | 11 个普通部门 | + 发布本部门任务/公告/票务、生成本部门邀请码 |
| 2 | 主席团成员 | `presidium` | 主席主任团 | + 跨部门发布、查看所有成员 |
| 3 | 主席 | `president` | 主席主任团 | + 人员裁撤、人员部门调动 |
| 3 | 老师 | `teacher` | 无 | 同上（最高权限） |

- 主席和老师权限同级（level 3）
- `president` 和 `teacher` 共享最高权限判断，代码中用 `hasMinRole(user.role, 'president')` 判断
- 老师无部门归属，department 字段可为空或填特殊值

## 四、邀请码改造

### 数据库变更

`invite_codes` 表新增 `role` 字段：

```sql
ALTER TABLE invite_codes ADD COLUMN role TEXT NOT NULL DEFAULT 'volunteer';
```

邀请码结构变为：`code | department | role | is_used | used_by | created_at`

### 生成逻辑

现有管理页 `InviteCodeManage.tsx` 的生成表单需新增「角色」下拉框。
主席团成员 (presidium) 只能生成自己部门的邀请码；部门负责人只能生成本部门 `volunteer` 邀请码；主席/老师可生成所有部门所有角色的邀请码。

## 五、登录流程

### 学生入口（现有流程改进）

1. 输入姓名 + 学号 + 邀请码 → 验证
2. 邀请码有效 → 获取 `department` + `role`
3. 判断学号是否已注册：
   - 新用户 → 设置密码 → `signUp(name, studentId, code, password, department, role)`
   - 老用户 → 输入密码 → `signIn(studentId, password)`
4. 注册时角色直接从邀请码的 `role` 字段读取，不再硬编码 `volunteer`

### 教师入口（新增）

独立的登录流程，与学生入口在同一卡片内通过 Tab 切换：

1. 输入教师邀请码
2. 判断教师学号（工号）是否已注册：
   - 新教师 → 设置密码 → 注册为 `teacher` 角色
   - 老教师 → 输入密码 → 登录
3. 教师注册无需姓名/学号字段（或用工号替代）

## 八、不涉及变更的部分

- 任务模块、公告模块、论坛模块、抢票模块、个人中心模块——这些模块内部逻辑不变，只依赖 `hasMinRole()` 和 `useAuth()` 做权限判断
- 现有已注册用户的 role 字段不变（手动在数据库中更新即可）
- CSS Modules 隔离机制不变
- 部署流程不变

## 九、风险点

1. **邀请码 role 字段为空**：旧邀请码没有 role → SQL 添加 `DEFAULT 'volunteer'` 兜底
2. **教师无 department**：`getDepartmentLabel` 等函数需处理 department 为空的情况
3. **主席团成员 (presidium) vs 主席 (president)**：两者同在「主席主任团」部门但权限不同，`hasMinRole` 用 level 值区分即可
