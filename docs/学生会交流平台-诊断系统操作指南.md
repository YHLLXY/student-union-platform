## 一、系统概述

诊断系统是一套轻量的**结构化标签日志**基础设施，位于 `src/diagnostics/`。它的核心目标是：**当线上或开发中出现 bug 时，开发者搜一个标签就能直接定位到出问题的文件，无需遍历整个代码库。**

### 核心能力

| 能力 | 说明 |
|------|------|
| 错误精准定位 | 每个错误自动带 `[模块/文件]` 标签，搜标签即定位 |
| API 调用追踪 | 每个 Supabase 请求自动记录耗时和成功/失败状态 |
| 生产环境降级 | 开发模式输出全部日志，生产模式静默 debug 级别 |
| 扩展预留 | 接口化设计，后期可接入 Sentry 等外部监控 |

## 二、快速上手

### 2.1 在组件中使用 logger

```tsx
// 1. 导入 logger
import { logger } from '../../diagnostics';

// 2. 创建绑定了模块/文件标签的 logger 实例
const log = logger.for('tasks/TaskForm');

// 3. 在关键操作点使用
export default function TaskForm() {
  const handleSubmit = async (values) => {
    try {
      await taskService.createTask(values);
      log.info('任务创建成功', { title: values.title });          // 成功操作
    } catch (e) {
      log.error('提交任务失败', e);                               // 错误捕获
      message.error('提交失败，请重试');
    }
  };

  return ( /* ... */ );
}
```

### 2.2 在 Service 中使用 trace 包装器

```ts
// 1. 导入 trace
import { trace } from '../../diagnostics';

const TAG = 'tasks/taskService';

// 2. 用 trace() 包装每个对外函数
export const fetchTasks = trace(TAG, 'fetchTasks', async (dept: string) => {
  const { data, error } = await supabase.from('tasks').select('*').eq('department', dept);
  if (error) throw error;
  return data;
});

export const createTask = trace(TAG, 'createTask', async (input: TaskInput) => {
  const { data, error } = await supabase.from('tasks').insert(input).select('*').single();
  if (error) throw error;
  return data;
});
```

### 2.3 在现有 Service 中最小改动集成

如果不想改动函数签名（保持原有 `export async function` 模式），只需替换 `console.error`：

```ts
import { logger } from '../../diagnostics';
const log = logger.for('tasks/taskService');

// 原来：console.error('fetchTasks error:', error);
// 改为：
log.error('fetchTasks 查询失败', error);
```

## 三、日志级别与输出格式

### 3.1 四级日志

| 级别 | 方法 | 使用场景 | 生产环境 |
|------|------|---------|:--:|
| `debug` | `log.debug()` | 开发调试细节、组件生命周期 | ❌ 静默 |
| `info` | `log.info()` | 操作成功、API 调用完成 | ✅ 输出 |
| `warn` | `log.warn()` | 超时重试、降级处理、非致命异常 | ✅ 输出 |
| `error` | `log.error()` | 渲染崩溃、请求失败、数据异常 | ✅ 输出 |

### 3.2 输出格式

```
[级别] [模块/文件] 消息 { 附加数据 }
```

**实际示例：**

```
[INFO]  [tasks/TaskForm] 任务创建成功 {"title":"迎新晚会筹备"}
[ERROR] [tasks/TaskForm] 提交任务失败 { error: "duplicate key value", userId: "abc-123" }
[INFO]  [tasks/taskService] createTask 完成 { duration: 234ms }
[INFO]  [tasks/taskService] fetchTasks 失败 { duration: 1200ms }
[ERROR] [tasks/taskService] fetchTasks 执行失败 { error: "Request timeout" }
[WARN]  [notices/NoticeForm] 响应超时，正在重试
[ERROR] [tasks/TaskListPage] 模块渲染错误 { error: "Cannot read properties of null" }
[ERROR] [app/ErrorBoundary] 全局渲染错误 { error: "ChunkLoadError" }
```

## 五、Bug 定位操作流程

### 场景一：用户反馈某个页面白屏

1. 打开浏览器控制台（F12 → Console）
2. 搜索 `[ERROR]`
3. 找到标签，例如 `[ERROR] [tasks/TaskListPage]`
4. 根据标签速查表定位文件：`src/modules/tasks/TaskListPage.tsx`
5. 查看错误消息定位具体问题

### 场景二：某个操作提交失败

1. 打开控制台，搜索操作相关的关键词（如 `createTask`）
2. 找到 API 调用日志：
   ```
   [INFO] [tasks/taskService] createTask 失败 { duration: 1200ms }
   [ERROR] [tasks/taskService] createTask 执行失败 { error: "Request timeout" }
   ```
3. 确认是网络超时问题（duration 1200ms），而非代码逻辑错误
4. 检查 Supabase 连接状态或网络环境

### 场景三：全局崩溃

1. 控制台搜索 `[app/ErrorBoundary]`
2. 查看错误消息和堆栈：
   ```
   [ERROR] [app/ErrorBoundary] 全局渲染错误 { error: "ChunkLoadError" }
   ```
3. 判断是代码分块加载失败 → 检查 Vite 构建配置或 CDN 部署

### 定位技巧

| 搜什么 | 找到什么 |
|--------|---------|
| `[ERROR]` | 所有错误 |
| `[ERROR] [tasks/` | 任务模块的所有错误 |
| `[taskService]` | 所有 service 层 API 调用记录 |
| `失败` | 所有失败的 API 调用 |
| `duration` | 所有 API 调用耗时 |

## 六、新模块集成步骤

以后新增功能模块时，按以下步骤接入诊断系统：

### Step 1：在 MODULE_TAGS 中添加标签

编辑 `src/utils/constants.ts`，在 `MODULE_TAGS` 中新增条目：

```ts
export const MODULE_TAGS = {
  // ... 已有模块 ...
  chat: {
    ChatRoom: 'chat/ChatRoom',
    ChatInput: 'chat/ChatInput',
    chatService: 'chat/chatService',
  },
} as const;
```

### Step 2：Service 层接入

```ts
// src/modules/chat/chatService.ts
import { logger } from '../../diagnostics';
import { trace } from '../../diagnostics';

const TAG = 'chat/chatService';
const log = logger.for(TAG);

export const sendMessage = trace(TAG, 'sendMessage', async (data: MessageInput) => {
  const { error } = await supabase.from('messages').insert(data);
  if (error) throw error;
});
```

### Step 3：组件层接入

```tsx
// src/modules/chat/ChatRoom.tsx
import { logger } from '../../diagnostics';
const log = logger.for('chat/ChatRoom');
```

## 七、配置与调试

### 7.1 日志级别控制

配置文件：`src/diagnostics/config.ts`

```ts
export const diagConfig: DiagConfig = {
  level: import.meta.env.DEV ? 'debug' : 'info',
  target: 'console',
};
```

- `npm run dev` → 所有级别输出（debug + info + warn + error）
- `npm run build` → 仅 info 及以上（debug 静默）

### 7.2 临时在生产环境开启 debug

修改 `config.ts`，将 `level` 改为 `'debug'`：

```ts
level: 'debug',  // 强制输出全部日志
```

重新构建部署即可。

### 7.3 查看最近错误

在浏览器控制台中执行：

```js
// 需要在代码中暴露 getRecentErrors（可在 main.tsx 中临时添加 window.__diag = { getRecentErrors }）
```

或者直接查看 errorReporter 的内存缓冲（通过 `src/diagnostics/errorReporter.ts` 的 `getRecentErrors()`）。

## 八、扩展指南

诊断系统预留了以下扩展点，后期可按需启用：

### 8.1 接入 Sentry 外部监控

当项目上线需要远程错误追踪时：

1. 安装 Sentry SDK
2. 修改 `src/diagnostics/errorReporter.ts`，将 `consoleErrorReporter` 替换为 Sentry 实现：

```ts
import * as Sentry from '@sentry/react';

export const sentryErrorReporter: IErrorReporter = {
  report(error, tag, context) {
    Sentry.captureException(error, {
      tags: { module: tag },
      extra: context,
    });
  },
};

// 在 main.tsx 中：
initErrorReporter(sentryErrorReporter);
```

**只改 1 个文件**，其余代码零变动。

### 8.2 添加开发者调试面板

1. 在 `src/diagnostics/` 下创建 `DebugPanel.tsx`
2. 使用 `getRecentErrors()` 获取错误列表
3. 在 `App.tsx` 中添加路由（仅 dev 模式可见）

### 8.3 日志持久化

修改 `config.ts` 的 `target` 为 `'storage'`，在 `logger.ts` 中添加 localStorage 写入逻辑。

### 8.4 网络请求全局拦截

在 `serviceWrapper.ts` 中扩展 `traceAll()` 函数，自动包装所有 fetch 请求。

## 十、注意事项

1. **标签不要硬编码**：优先使用 `MODULE_TAGS` 常量中的标签引用，保持一致性
2. **不要吞错误**：`log.error()` 后仍要合理处理错误（显示 Toast、降级 UI 等），logger 只负责记录，不负责处理
3. **生产环境注意敏感信息**：不要在 `log.info()` 的 data 参数中传入密码、token 等敏感数据
4. **debug 日志要节制**：不要在循环中大量输出 debug 日志，避免控制台性能问题
5. **trace() 不重试**：`trace()` 只追踪不重试，重试逻辑应在 service 自身实现
