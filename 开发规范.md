## 一、最高优先级准则

### 1.1 禁止私自扩展修改范围

**【铁律】** 任何任务执行时，严格按需求范围修改，**不得私自扩大修改面**。

- ✅ 任务要求改 A 文件的 B 函数 → 只改 A 文件的 B 函数
- ❌ 顺手把 A 文件的 C 函数也重构了
- ❌ 发现另一个文件有类似问题就直接去改

**如果发现关联问题或改进空间**：汇报但不私自动手，等待确认后再执行。

### 1.2 避免次生问题

每次修改前问自己：

1. 这个改动会影响哪些其他模块？
2. 会不会破坏已有的功能？
3. 修改后是否需要同步更新类型定义、路由配置、样式文件？

**原则**：改动越少，次生问题越少。优先选择对现有代码侵入最小的方案。

### 1.3 任务执行前评估可行性

在执行任何非平凡任务前：

1. 确认理解需求的完整意图
2. 评估实现方案的可行性和风险
3. 预估影响范围（涉及哪些文件、哪些模块）
4. 发现潜在问题及时汇报，不盲目执行

## 二、模块化开发规范

### 2.1 模块目录结构

```
src/modules/<模块名>/
├── index.tsx           ← 对外唯一入口（路由引用此文件）
├── <Component>.tsx      ← 模块内部组件
├── <module>Service.ts   ← 数据服务层（Supabase 调用封装）
└── <module>.module.css  ← CSS Modules 样式（强制隔离）
```

### 2.2 模块隔离原则

| 规则 | 说明 |
|------|------|
| **只暴露 index.tsx** | 其他模块只能 import 入口文件，不可直接引用内部组件 |
| **Service 不对外暴露** | 数据服务层仅模块内部使用 |
| **样式强制隔离** | 必须使用 CSS Modules（`*.module.css`），禁止全局样式污染 |
| **独立错误边界** | 每个模块包裹 `ModuleErrorBoundary`，崩溃不影响其他模块 |
| **新增模块不改旧代码** | 只需加路由配置和菜单项，已有模块代码零改动 |

### 2.3 模块间通信

```
✅ 允许：
  - 通过共享组件（src/components/）传递数据
  - 通过 AuthContext 获取用户信息
  - 通过 Supabase 客户端共享数据层

❌ 禁止：
  - 模块 A 直接 import 模块 B 的内部组件
  - 模块间相互修改对方的 state
  - 跨模块的全局变量共享
```

### 2.4 共享资源

| 目录 | 用途 | 规则 |
|------|------|------|
| `src/components/` | 通用 UI 组件（ErrorBoundary、AppLayout 等） | 不含业务逻辑 |
| `src/utils/` | 常量、工具函数 | 纯函数，不含副作用 |
| `src/diagnostics/` | 诊断基础设施 | 不依赖任何业务模块 |

## 三、代码组织规范

### 3.1 文件职责单一

- 每个文件只做一件事
- 超过 300 行的组件考虑拆分
- Service 文件只包含 Supabase 调用逻辑，不含 UI 逻辑

### 3.2 导入顺序

```tsx
// 1. React 核心
import { useState, useEffect } from 'react';

// 2. 第三方库
import { Button, message } from 'antd';
import { ReloadOutlined } from '@ant-design/icons';

// 3. 项目内部（diagnostics / components / utils）
import { logger } from '../../diagnostics';
import AppLayout from '../../components/AppLayout';

// 4. 模块内部
import { taskService } from './taskService';

// 5. 样式
import styles from './tasks.module.css';
```

### 3.3 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 组件文件 | PascalCase | `TaskForm.tsx` |
| Service 文件 | camelCase + Service | `taskService.ts` |
| CSS Module | camelCase + .module.css | `tasks.module.css` |
| 函数/变量 | camelCase | `fetchTasks`、`userName` |
| 常量 | UPPER_SNAKE_CASE | `MODULE_TAGS`、`ROLES` |
| 类型/接口 | PascalCase | `Task`、`ILogger` |
| 路由路径 | kebab-case | `/task-review` |

### 3.4 TypeScript 规范

- 所有函数导出必须有明确类型注解
- 禁止使用 `any`，不确定类型时使用 `unknown`
- 接口优先于 type（除非需要 union/intersection）
- Service 函数的入参和返回值必须定义类型

## 四、错误处理规范

### 4.1 三层错误架构（不可降级）

```
层级 1: ErrorBoundary（全局）
  └── 捕获未预料的 React 渲染错误 → 显示友好降级页

层级 2: ModuleErrorBoundary（模块级）
  └── 模块崩溃时仅该区域降级 → 其余模块正常运行

层级 3: Service try-catch（数据层）
  └── 网络/权限/业务错误 → Toast 提示 + 结构化日志
```

### 4.2 错误处理规则

- **不要吞错误**：catch 之后至少要输出日志，不能空 catch
- **用户友好**：错误提示用中文，说明问题和建议操作
- **开发可见**：所有错误必须通过 `logger.error()` 输出到控制台
- **不影响其他模块**：一个模块的错误不能导致其他模块不可用

### 4.3 边界情况必须处理

| 场景 | 处理方式 |
|------|---------|
| 列表为空 | 显示 Ant Design Empty 组件 + 引导文字 |
| 网络断开 | 检测 `navigator.onLine`，显示离线提示条 |
| 登录过期 | 拦截 401，自动跳转登录页 |
| 加载中 | Ant Design Spin 组件 |
| 并发操作 | 数据库事务 + 行锁 |

## 五、诊断调试规范

### 5.1 必须使用结构化日志

禁止直接用 `console.error/warn/log` —— 必须通过诊断系统的 `logger`：

```tsx
// ❌ 禁止
console.error('fetchTasks error:', error);

// ✅ 必须
import { logger } from '../../diagnostics';
const log = logger.for('模块/文件');
log.error('fetchTasks 查询失败', error);
```

### 5.2 标签 = 模块名/文件名

标签格式严格遵循 `<模块名>/<文件名>`，标签常量统一在 `MODULE_TAGS` 中管理。

### 5.3 Service 层 API 调用必须追踪

每个对外 service 函数必须使用 `trace()` 包装或用 `logger` 记录：

```ts
export const createTask = trace('tasks/taskService', 'createTask', async (data) => { ... });
```

### 5.4 日志级别选择

| 场景 | 级别 |
|------|------|
| 开发调试详情 | `debug` |
| 操作成功、API 完成 | `info` |
| 超时重试、降级处理 | `warn` |
| 渲染崩溃、请求失败 | `error` |

## 六、AI 协作规范

### 6.1 文件读取

- **拒绝无意义重复读取**：同一文件在同一会话中只读取一次，后续引用已有内容
- **精细化读取**：读取所需范围，不全文加载不需要的代码
- **读取前先确认**：确认是否已有该文件的上下文，避免重复

### 6.2 任务执行

- 执行前检查：需求理解确认 → 可行性评估 → 影响范围预估 → 风险评估
- 执行中遵守：不私自扩范围 → 不跳过验证步骤 → 有问题及时汇报
- 执行后验证：TypeScript 编译 → 构建验证 → 功能逻辑检查

### 6.3 代码修改

- 先设计方案、后编写代码（禁止跳过设计直接写代码）
- 最小化改动原则：能用 2 行解决的不用 20 行
- 遵循现有代码风格（不引入新的编码习惯）
- 修改完成后必须通过 `tsc --noEmit` 验证

### 6.4 提交规范

- 每个功能点独立 commit
- Commit message 格式：`feat:` / `fix:` / `docs:` / `refactor:` + 中文描述
- 不混入无关文件的改动

## 七、技术栈约束

| 层面 | 技术 | 版本 | 约束 |
|------|------|------|------|
| 前端框架 | React | 19.x | 函数组件 + Hooks，class 组件仅限 ErrorBoundary |
| 构建工具 | Vite | 6.x | 使用 `import.meta.env.DEV` 判断环境 |
| UI 组件库 | Ant Design | 5.x | 使用 antd 原生组件，不自造轮子 |
| 样式方案 | CSS Modules | — | 文件命名 `*.module.css` |
| 路由 | React Router | 7.x | HashRouter（GitHub Pages 兼容） |
| 后端服务 | Supabase | — | 所有数据操作通过 Supabase Client |
| 语言 | TypeScript | — | 严格模式，禁止 `any` |

## 八、禁止事项清单

| 序号 | 禁止行为 | 原因 |
|:--:|------|------|
| 1 | 私自扩展任务修改范围 | 最高优先级铁律 |
| 2 | 跳过设计直接写代码 | 导致返工和次生问题 |
| 3 | 重复读取同一文件 | 浪费上下文和 token |
| 4 | 使用 `console.error` 裸输出 | 破坏诊断标签体系 |
| 5 | 修改模块内部文件不更新标签 | 破坏定位能力 |
| 6 | 跨模块 import 内部组件 | 破坏模块隔离 |
| 7 | 使用全局 CSS 样式 | 样式污染 |
| 8 | 空 catch 块吞错误 | 问题不可见 |
| 9 | 在循环中大量输出 debug 日志 | 控制台性能问题 |
| 10 | 在日志中输出密码/token | 安全风险 |
